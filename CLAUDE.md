# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Commands

```bash
# Development (HMR via electron-vite)
npm run dev

# Production build (outputs to out/)
npm run build

# Package into installer (dist/)
npm run dist

# Recompile native modules (node-pty, better-sqlite3) after electron version changes
npm run rebuild

# Type-check both projects (web + node) ‚Äî the correctness gate
npm run typecheck
```

**Risky pure logic is extracted into dependency-free modules specifically so it can be unit-tested** ‚Äî renderer logic under `src/lib/` (e.g. `lineDiff.ts` ‚Äî the Composer's LCS diff; `mention.ts` ‚Äî the @-mention caret parser; `snippets.ts` ‚Äî snippet CRUD/parse) and main-process logic under `electron/lib/` (e.g. `aiParse.ts` ‚Äî `stripCodeFences`/`extractJsonObject`, the shared defensive parsing every LLM-response handler uses). These import nothing from React/CodeMirror/Electron, so they strip-and-run straight from source with no build step. `npm test` (`node --experimental-strip-types tests/lib-units.mts`, Node 22+) runs the committed assertions for all of the above ‚Äî **add a block to `tests/lib-units.mts` whenever you add a new pure module under `src/lib/` or `electron/lib/`**, and keep it green. (The other `tests/*.mjs` ‚Äî `smoke`, `editor-verify` ‚Äî are Playwright integration checks that boot the built app; this is the no-build unit layer.) Prefer this pattern for new algorithmic code: keep it pure, in a `lib/`, and assert against it. TypeScript type-checking remains the primary correctness gate, and `npm run typecheck` (runs `tsc --noEmit` over `tsconfig.web.json` and `tsconfig.node.json`) is currently **clean** ‚Äî keep it that way. The build (`electron-vite`/esbuild) does **not** gate on type errors, so run `npm run typecheck` before declaring work done. Note: `tsc` is `composite`, so a stale `*.tsbuildinfo` can mask the effect of new `.d.ts` files ‚Äî delete `tsconfig.web.tsbuildinfo` if an augmentation isn't taking.

## Architecture

SwarmMind is an Electron desktop app that runs multiple AI coding CLIs side-by-side in resizable terminal panes, with a shared MCP memory server so agents can exchange context.

### Process boundary

The renderer (React) and main process (Node.js) are strictly isolated via contextBridge:
- `electron/preload.ts` ‚Äî exposes the entire IPC surface as `window.swarmmind`
- `src/types/swarmmind.d.ts` ‚Äî TypeScript types for `window.swarmmind`

**Never call Node.js APIs directly from React code.** Every cross-process call must go through `window.swarmmind.*`.

### Dual SQLite databases

Two independent `better-sqlite3` connections live in `memory/db.ts`:

| Connection | Path | Tables |
|---|---|---|
| `appDb` | `userData/app.db` | `workspaces`, `skills`, `app_state` |
| `workspaceDb` | `{rootPath}/.swarmmind/memory.db` | `memory_entries`, `tasks`, `pane_layouts`, `agent_configs`, `messages` |

`initAppDb()` runs once at startup. `initWorkspaceDb()` runs each time the user opens a workspace (the previous connection is closed first). All query helpers in `memory/queries.ts` call `getAppDb()` or `getWorkspaceDb()` ‚Äî never the wrong one.

### MCP server

`mcp/server.ts` runs an Express/SSE HTTP server inside the main process (preferred port 57400, falls back to OS-assigned). Tools and resources are registered in `mcp/tools.ts` and `mcp/resources.ts`. The active workspace ID is tracked via `setActiveWorkspace()` so tools always query the correct DB.

**Auth:** the SSE endpoint is gated on a bearer token derived per workspace (`getMcpToken(workspaceId)` = HMAC of a per-run master secret over the workspace id). It's appended to the injected SSE URL (`?token=‚Ä¶&ws=‚Ä¶`) and exposed as `SWARMMIND_MCP_TOKEN`, so only agents SwarmMind spawns can reach the server even though it binds `127.0.0.1`. `/mcp/sse` requires the token to match the one derived for the requested `ws` (constant-time compare), so a token issued for one workspace can't be replayed against another by swapping `?ws=` ‚Äî each token authorizes only its own workspace's DB.

**Tools:** `memory_read/write/delete/list`; `task_create/update/list/get/note`; `send_message`. `task_note` appends a timestamped progress note (status unchanged); `task_get` returns full detail; `send_message(to,from,message)` queues a directed agent‚Üíagent message in the `messages` table, which the renderer-side conductor delivers by injecting into a free running pane of the recipient agent (see `useConductor.ts::deliverMessages`, runs every tick regardless of orchestration mode).

At agent spawn time, `electron/pty-manager.ts::injectMcpConfig()` writes the MCP SSE URL (with token) into each agent's config file (e.g., `{cwd}/.claude/settings.json` for Claude Code).

**Secrets:** agent API keys are encrypted at rest via Electron `safeStorage` (`electron/secrets.ts`), wrapped by `electron/agent-config.ts::readAgentConfig/writeAgentConfig` (settings IPC reads/writes go through these). Legacy plaintext keys are read unchanged and re-encrypted on next write; if the OS keychain is unavailable, it falls back to plaintext rather than failing the save.

**Untrusted-workspace spawn safety:** `agent_configs` lives in the per-repo workspace DB (`{rootPath}/.swarmmind/memory.db`), so opening a cloned repo loads attacker-controllable config, and the spawn-affecting fields (`executablePath`/`extraFlags`/`env`) feed the launched shell command ‚Äî a zero-click RCE if honored blindly. `writeAgentConfig` signs those three fields with a per-install HMAC key kept in `app.db` (userData, never in a repo), and the spawn path reads via `readAgentConfigForSpawn`, which **drops** them unless the signature verifies. So values configured through *this* install's Settings work; values shipped inside a repo are ignored (re-save in Settings to trust them). Defense-in-depth: `pty-manager.ts::buildShellCommand`/`quoteForShell` shell-quote every argv token (PowerShell single-quotes + call operator, bash single-quotes, cmd caret-escapes metachars) since `node-pty` must wrap the command in a shell on Windows.

### PTY spawning (Windows critical path)

`node-pty` cannot spawn `.cmd` scripts directly on Windows. `resolveSpawn()` in `electron/pty-manager.ts` wraps every agent command in the user's selected shell:

- `powershell` ‚Üí `powershell.exe -NoLogo -NoExit -Command <cmd>`
- `cmd` ‚Üí `cmd.exe /k <cmd>`
- `bash` ‚Üí `bash.exe -i -c <cmd>` (Git Bash or WSL's bash, whichever is on PATH)

On non-Windows the command is wrapped in `$SHELL -c <cmd>` so quoted args/paths survive. The shell style (`ShellStyle = 'powershell' | 'cmd' | 'bash'`) is stored in Zustand (`shellStyle`) and persisted via `appsetting:set`. Users configure it in Settings ‚Üí General.

### Session resume

Reopening a workspace automatically relaunches each pane's agent **and** restores its prior conversation ‚Äî no user action. The mechanism:

- When an agent spawns, the pane leaf records `agentRunning: true` and a `sessionId` (a UUID minted on fresh spawn). Both are persisted in the layout JSON. `killAll()` kills **silently** (suppresses `pty:exit`) so quitting/workspace-switch doesn't clear those flags.
- On load, `sanitizeLoadedLayout()` in `workspace.ts` turns any `agentRunning` leaf into a queued auto-spawn (`pendingAutoSpawn: true`, `resume: true`) instead of wiping it, preserving `agentId`, `cwd`, and `sessionId`.
- `ptyCreate(..., resume, sessionId)` resolves the launch via `buildLaunchArgs()` in `pty-manager.ts`. Claude uses a caller-chosen session id for **per-pane** continuity: fresh = `claude --session-id <uuid>`, resume = `claude --resume <uuid>` (both non-interactive ‚Äî no picker). Agents without a controllable session id fall back to `AGENT_RESUME_ARGS` ("continue last", e.g. `codex resume --last`, `opencode --continue`); the rest relaunch fresh.

**StrictMode gotcha (the auto-spawn effect in `AgentPane.tsx`):** the deferred-spawn timer must **not** be cancelled in the effect cleanup. React StrictMode runs effects mount‚Üícleanup‚Üímount in dev; a `clearTimeout` in cleanup kills the only scheduled spawn while the `autoSpawnedRef` guard blocks rescheduling, so the agent never launches (blank pane). Guard the timer callback with a mount ref instead. This only bites in `npm run dev`, not packaged builds.

### Multi-pane orchestration & extras

Features layered on top of the pane model:

- **Broadcast & selection** (`BroadcastBar.tsx`, store `selectedPaneIds`/`broadcastBarOpen`): Ctrl/‚åò-click a pane title bar to select it; the broadcast bar (TopBar button or Ctrl/‚åò-B) sends one prompt to selected panes, or all when none are selected, via `window.swarmmind.ptyInput`.
- **Pipe** (AgentPane context menu): route a pane's selection (or recent output, ANSI-stripped via `usePty.getRecentOutput`) to shared memory (`memoryWrite`) or into the other panes.
- **Agent activity** (`pty-manager` ‚Üí `pty:state` event ‚Üí store `paneAttention`): a per-pane idle timer flips agents between `working` and `waiting`; this drives the pane "waiting" badge and the conductor, and is purely about "finished a turn" ‚Äî it does **not** notify. **Notifications are question-gated:** going `waiting` only fires the discrete "needs you" signal when the agent is actually blocked on an answer. `attachPty` keeps a rolling, ANSI-stripped `PtyEntry.recentOutput` tail and, on going quiet, runs `looksLikeQuestion()` (permission prompt / `y/n` / selection-menu patterns) against it. Only on a match does it (a) send `pty:attention` ‚Üí renderer `addPaneNotification` (the TopBar bell, regardless of focus) and (b) fire an OS `Notification` (only if the window is unfocused). Both still require `PtyEntry.hadInput` (set in `ptyInput`) so a freshly-spawned agent at its prompt never pings. The idle threshold is the module-level `agentIdleMs` (default 4000), configurable in Settings ‚Üí General (persisted as the `agentIdleMs` app setting; `appsettings.ts::applySideEffects` pushes changes live to `pty-manager.ts::setAgentIdleMs`, and `loadPersistedSettings()` applies it at startup). Cleared on focus/exit.
- **Session history & scrollback** (`electron/ipc/sessions.ts`): `session:list` reads `~/.claude/projects/<encoded-cwd>/*.jsonl` for the Claude session picker (`SessionPicker.tsx`); `scrollback:load/save` persist per-pane terminal output under `{root}/.swarmmind/scrollback/<paneId>.log` (debounced from `usePty`).
- **Command palette** (`CommandPalette.tsx`, store `commandPaletteOpen`, Ctrl/‚åò-K) and **terminal search** (`@xterm/addon-search` in `usePty`, Ctrl-F per pane).
- **Per-pane title/colour**: `PaneLeaf.title`/`color` (persisted), edited via title-bar double-click and the context-menu swatches.
- **Orchestration ‚Äî Conductor + Lead** (`hooks/useConductor.ts`, `OrchestratorBar.tsx`, store fields prefixed `orchestrat*`/`leadPaneId`/`paneTask`): turns the existing task queue + the `pty:state` `waiting` signal into an autonomous control loop. `useConductor()` is mounted once in `App.tsx`; its tick is an **idempotent reconciler scheduled event-driven** ‚Äî woken by swarm-bus events (pure `isWakeEvent` gate in `conductor.ts`: task/message/memory/pane changes; the conductor's own `dispatch`/`synthesis` output and telemetry types are excluded to avoid echo) and by store changes (pane attention/tree, orchestration controls), coalesced through a 150ms debounce, with a 5s heartbeat kept **only** for the time-based watchdogs (stall nudge, decompose timeout). It spends **zero** model tokens ‚Äî code does the wiring, panes do the thinking. Three modes (`orchestrationMode`: `off`/`assisted`/`auto`, toggled in `OrchestratorBar`, shown as a TopBar dot). Each tick: (1) **completion sweep** ‚Äî a worker reports done by calling `task_update(id,'done')` + `memory_write('result:<id>',‚Ä¶)` over MCP; the loop watches `taskList`, collects the result, and frees the pane (`paneTask` maps paneId‚ÜítaskId); (2) **dispatch** ‚Äî finds the next `pending` task whose `depends_on` (new comma-separated `tasks.depends_on` column + `task_create` MCP arg) are all `done`, matches it to a free worker pane by `assigned_agent`, and injects a single-line prompt via `ptyInput` (in `assisted` mode it raises an `orchestratorProposal` for `conductorControls.approve()`/`skip()` instead); (3) a designated **lead pane** (`leadPaneId`, optional) is prompted to **decompose** `orchestratorGoal` into tasks at run start and to **synthesize** results once all tasks finish (`orchestratorPhase`: `idle`‚Üí`running`‚Üí`done`). Worker/lead panes must already be running their CLIs ‚Äî the conductor injects prompts, it does not spawn agents. **Robustness** (session-scoped refs, reset on each run): a `failed` task is auto-reset to `pending` and re-dispatched up to `MAX_RETRIES` before being surfaced as needing attention; a dispatched worker that goes idle (`paneAttention === 'waiting'`) past `STALL_MS` while its task is still `in_progress` gets one nudge to report completion; if the lead produces no tasks within `DECOMPOSE_TIMEOUT_MS` it's re-prompted once, then the loop gives up rather than hanging. **Every per-tick decision is a pure function in `src/lib/conductor.ts`** (sweep/watchdog/dispatch/review/synthesis/message routing + the prompt builders, unit-tested in `tests/lib-units.mts`); the hook is only the impure shell ‚Äî change orchestration behaviour there, not inline in the hook. The activity log is persisted per workspace (debounced app setting `orchestratorLog:<id>`, loaded on workspace switch).

Actions: **Commit all** (stage + commit inside the worktree so the work becomes mergeable), **Merge into `<base>`** (merges committed work; aborts and reports on conflict so the main checkout stays clean), **Discard** (remove worktree + delete branch). - **Snippets** (`src/lib/snippets.ts`, store `snippets`, `FileEditor.tsx` status-bar **‚ùè Snippets**): reusable code fragments saved from a selection and inserted at the cursor ‚Äî not AI, so fully testable offline. The dropdown lists saved snippets (click to insert, √ó to delete) and offers "Save selection‚Ä¶" (enabled when text is selected) with an inline name field. Persisted globally as the `editorSnippets` JSON app-setting (loaded in `App.tsx`); pure CRUD/filter/parse logic in `src/lib/snippets.ts` is unit-tested. Note: distinct from `SkillsLibrary` (drag-drop *prompts* for agent panes) ‚Äî snippets are editor text.
- **Rename symbol across files ‚Äî F2** (`FileEditor.tsx`, store `composerSeed`/`openComposerWith`): press **F2** on an identifier ‚Üí a small popover prefilled with the current name; entering a new name greps the workspace for usages (`fsSearchFiles`), then routes a precise rename instruction + the matched files (plus the current file, capped at 12) through the **Composer** via `openComposerWith({instruction, contextPaths})`. The Composer consumes the one-shot `composerSeed` on open (prefills the instruction, loads the context files) and clears it ‚Äî so rename reuses the Composer's preview/checkpoint/apply pipeline rather than adding new infra. Identifier-guarded (won't fire on non-words); the instruction tells the model not to touch substring matches, strings or comments.
- **AI diagnostics / "Fix with AI"** (`FileEditor.tsx`, backend `swarmAgent:diagnose`, `@codemirror/lint`): the status-bar **‚öÝ Diagnose** button sends the open file (line-numbered) to the model, which returns strict-JSON problems `[{line, severity, message, fix}]`. These are mapped to CodeMirror `Diagnostic`s and rendered in `lintGutter()` (underlines + gutter markers); each carries a **"Fix with AI"** action that opens the Cmd-K inline-edit widget on the flagged line prefilled with the suggested fix ‚Äî i.e. diagnostics feed straight back into the inline-edit flow. The button shows analyzing / "‚úì No issues" / "‚öÝ N issues"; diagnostics are cleared on file switch.
- **Ghost-text autocomplete ‚Äî Tab** (`src/editor/autocomplete.ts`, backend `swarmAgent:complete`, store `ghostTextEnabled`): Copilot/Cursor-style inline AI completions. A `ViewPlugin` debounces (350ms) on typing, sends prefix/suffix around the cursor to a fast non-streaming Groq call, and shows the prediction as dimmed ghost-text via a widget decoration + `setGhost` `StateField`; **Tab** accepts (high-precedence keymap, falls through when no ghost), **Escape** or any edit dismisses. Stale responses are dropped by a request counter. **Off by default** (it spends tokens on every typing pause) ‚Äî toggled from the editor status-bar **‚ä≥ Autocomplete** button, persisted as the `editorGhostText` app setting (loaded in `App.tsx`). The plugin reads language + enabled state via refs so it's never rebuilt.
- **Inline AI edit ‚Äî Cmd/Ctrl+K** (`FileEditor.tsx` + `src/editor/inlineEdit.ts`, backend `swarmAgent:editCode`): the Cursor-style "vibe coding" surface inside the CodeMirror editor. Select code (or place the cursor for a pure insertion) and press **Ctrl/‚åò-K** (or the status-bar **‚ú¶ Edit with AI** button) ‚Üí a floating prompt appears at the selection. Typing an instruction calls a focused, single-turn Groq completion (reusing the SwarmAgent key/model, but with `EDIT_SYSTEM_PROMPT` that returns raw code only ‚Äî a defensive fence-strip handles models that wrap output) seeded with the selection plus ¬±`CONTEXT_LINES` of surrounding context. The prompt supports **@-mention file context**: typing `@` opens a fuzzy file-path autocomplete (from `fsListFiles`, warmed once per editor mount) ‚Äî picked files are read (`fsReadFile`, capped/sliced) and sent to the model as "Referenced files" so it can match a style or reuse a helper. The result streams into the widget, then replaces the selection and enters an **accept (‚èé) / reject (Esc) / regenerate** preview, with the changed range tinted via a CodeMirror decoration `StateField` (`setEditHighlight`). Reject restores the original text; a cancel-while-streaming guard (`activeReqRef`) drops a late result so it's never applied. **Keybinding note:** Ctrl/‚åò-K is also the global Command Palette shortcut; the editor's keymap (`Prec.highest`) handles it first, and `App.tsx`'s global handler skips `command-palette` when the event target is inside `.cm-editor`, so the same key means "inline edit" in the editor and "palette" everywhere else (mirroring Cursor).
- **Next-edit prediction ‚Äî "Tab to jump"** (`FileEditor.tsx` + pure `src/lib/nextEdit.ts`, backend `swarmAgent:nextEdit`): Cursor's tab-to-jump. After an inline AI edit is **accepted**, `acceptInlineEdit` fires a best-effort `swarmAgentNextEdit` call (the just-edited line range + full numbered source) to a non-streaming Groq completion (`NEXT_EDIT_SYSTEM_PROMPT`, strict JSON `{line,instruction}` or `{none:true}`). The renderer hands the raw prediction to `resolveNextEditTarget(pred, doc.lines, justEditedLine)` ‚Äî the pure, unit-tested validator that clamps the line into the document and rejects jumping back onto the line just edited ‚Äî and on a valid target shows a floating **Tab** chip (bottom-right of the editor) describing the follow-up. Pressing **Tab** (or clicking the chip) calls `jumpToNextEdit`: it selects the target line and reopens the inline-edit widget prefilled with the suggested instruction, so accepts chain into the next edit. The Tab/Escape handlers live in the same `Prec.highest` keymap as F2 and **fall through** (return false) when there's no hint, so ghost-accept and default Tab still work; the once-built keymap reads live state via `jumpRef`/`editRef`/`hintRef`. The hint is cleared on file switch and on any manual doc edit.
- **One-click apply from chat** (`SwarmAgentChat.tsx` `ApplyFromChat` + pure `src/lib/codeBlocks.ts`): a completed SwarmAgent reply that contains **file-targeted** fenced code blocks (path on the fence info string ‚Äî ```` ```ts src/foo.ts ````, ```` ```ts:src/foo.ts ````, `title=`/`file=` ‚Äî or on the line just before the fence ‚Äî `` `src/foo.ts` ``, `**src/foo.ts**`, `File: ‚Ä¶`) shows a **Review & apply** button. `extractFileBlocks` (pure, unit-tested) pulls `{path,language,content}` out of the markdown; the button calls `openComposerWith({ instruction, contextPaths:[], plan:{summary,changes} })` with a **pre-built plan** so the Composer skips the model round-trip and renders the diff/checkpoint/apply UI directly against the exact blocks. The Composer's seed effect branches on `seed.plan`: it seeds `changes`/`summary`/`selected`, resolves each file's current content for diffing (`''` for create/unreadable), and jumps to `phase:'result'`. Only rendered in the main window's chat (not the workspace-less widget).
- **Composer auto-context ‚Äî "Suggest relevant"** (`ComposerPanel.tsx` + pure `src/lib/retrieval.ts`): the first, model-free step toward automatic right-context selection (Cursor's strength). Instead of hand-picking context files, the **‚ú¶ Suggest relevant** button tokenizes the instruction (camelCase/snake_case-split, noise-dropped via `tokenize`), greps the workspace for its salient terms (`fsSearchFiles`, bounded), groups the matches by file, then **BM25-ranks** those files by their matched-line snippets (`rankDocs`) and adds the top few to the context chips. Bounded by construction ‚Äî only files that grep-match a query term are ever read. Pure ranker (`retrieval.ts`) is unit-tested. Ranking is **hybrid**: BM25 always, fused (reciprocal-rank fusion, `fuseRankings`) with on-device **embedding** similarity (`rankByEmbedding`/`cosineSim`) once the local model is warm ‚Äî see below.
- **On-device embeddings** (`src/lib/embed.ts`, pure math in `src/lib/retrieval.ts`): a key-free semantic layer for retrieval, mirroring SwarmVoice's transformers.js setup (`allowLocalModels=false`, the same `voiceCache*` IPC disk cache, locally served ONNX WASM from `public/ort/`). `embedTexts()` lazily loads `Xenova/all-MiniLM-L6-v2` (384-dim, ~23 MB quantized, mean-pooled + normalized) and embeds bounded text; `preloadEmbedder()`/`embeddingsReady()` let the UI warm it in the background so a click never blocks on the download. `embed.ts` is the **impure runtime**; the vector math it feeds (`cosineSim`/`rankByEmbedding`/`fuseRankings`) stays pure in `retrieval.ts` and is unit-tested. Composer auto-context uses it best-effort: when the model is ready it fuses BM25 + embedding rankings (RRF); otherwise it warms the model and uses lexical-only that round (offline-safe). Not imported by the test runner (only the pure `retrieval.ts` is).
- **Whole-repo semantic index** (`src/lib/codeIndex.ts`, pure deps `src/lib/chunk.ts` + `src/lib/retrieval.ts`): the step beyond grep-bounded auto-context ‚Äî instead of only ranking files that lexically match a query term, it ranks the **whole repo by meaning**. `buildIndex(rootPath)` lists indexable files (bounded by `MAX_FILES`/`MAX_CHUNKS`/`MAX_FILE_BYTES`), splits each into overlapping line windows (`chunkText`), embeds them in batches (`embedTexts`, on-device), and persists the vectors to `{root}/.swarmmind/vector-index.json` (`loadIndex` rejects a stale `version`/`model`). `queryIndex(entries, query, k)` embeds the query, scores every chunk (`rankByEmbedding`), then collapses to one row per file keeping its best chunk (`dedupeByPath`). Composer's **‚ú¶ Suggest relevant** uses it when an index exists (the panel's **Build index** button creates one, reporting embed progress), fusing those repo-wide hits with the grep/BM25 round; like `embed.ts` it's the **impure runtime** (fs + embeddings) and the math it leans on (`chunk.ts`, `retrieval.ts::rankByEmbedding`/`dedupeByPath`) is the unit-tested part.
- **Verify ‚Üí fix loop** (`ComposerPanel.tsx` + pure `src/lib/verify.ts` + constrained `electron/ipc/verify.ts`): the safe, human-in-the-loop core of "agent-mode iteration". After a Composer plan is **applied**, the panel offers to run one of the workspace's **own npm scripts** (picker defaults to typecheck/test/lint/build via `pickVerifyScript`/`orderVerifyScripts`); on a non-zero exit, **‚ú¶ Fix errors with AI** condenses the output (`summarizeFailure` ‚Äî error-like lines first) into a follow-up instruction (`buildFixInstruction`) and re-runs the Composer plan pipeline so the model fixes its own change. **Constrained runner safety:** `verify:run` executes **only** a script that exists in the repo's `package.json scripts` (allowlist) **and** matches a strict charset (`isSafeScriptName` ‚Äî no shell metacharacters), via `execFile` with no shell on POSIX (`npm`) and `shell:true` only on Windows where `npm.cmd` requires it ‚Äî so even a malicious cloned repo's `package.json` can't turn this into arbitrary command execution. Hard timeout + bounded output. Pure control logic (`verify.ts`) is unit-tested; the main-process copy of `isSafeScriptName` is duplicated (main can't import renderer paths) and must stay in sync. Every step is an explicit click ‚Äî no autonomous process execution.

### TypeScript language service (real LSP)

`electron/lsp/*` runs a **`ts.LanguageService` in a `worker_threads` worker** ‚Äî the last thing Cursor had that SwarmMind didn't. There is no external language server to install or spawn: the `typescript` package (a **runtime** dependency, not a dev one ‚Äî `externalizeDepsPlugin` only externalizes prod deps) *is* the engine. It covers TS/JS/TSX/JSX.

- **Why a worker:** `ts.LanguageService` is synchronous and the first program build takes ~1.2s. The main process pumps every PTY, so doing this on it would freeze all terminals. `electron/lsp/worker.ts` is a **second main-process entry** in `electron.vite.config.ts` (emitted as `out/main/lsp-worker.js`); `client.ts` resolves it as `__dirname/lsp-worker.js`, so those output names are load-bearing. The client degrades to "no result" on crash/timeout and respawns ‚Äî never throws into the editor.
- **Self-contained requests:** every query (`lsp:diagnostics/hover/definition`) ships the editor's live buffer, so the service always answers against unsaved text and a worker restart costs only the in-flight call (no resync handshake).
- **Program roots = the open files + the project's ambient `.d.ts` files.** TS pulls in the import closure itself, so we never enumerate the repo. **The `.d.ts` part is not optional:** ambient declarations are included by being *listed* in the project, not by being imported ‚Äî drop them and `window.swarmmind` becomes "Property 'swarmmind' does not exist on Window" and `import x from './y.png'` becomes "Cannot find module" on every file.
- **Which tsconfig owns a file** is resolved through **project references** (pure, unit-tested `chooseProject` in `electron/lib/tsLsp.ts`). Taking the *nearest* tsconfig is wrong in this very repo: the root `tsconfig.json` is solution-style (`{"files": [], "references": [‚Ä¶]}`), declares no files and no `jsx`, so trusting it flags every `.tsx` as broken. Roots are stored in their **real case** ‚Äî TS echoes root names back in definition results, and normalized (lowercased) names would break navigation on a case-sensitive filesystem.
- **The payoff ‚Äî compiler errors feed the AI fix loop.** `src/lib/diagnostics.ts::mergeDiagnostics` (pure, unit-tested) merges the language service's diagnostics with the model's (`swarmAgent:diagnose`) into one lint list: the compiler is authoritative (a TS *error* on a line suppresses an AI diagnostic there ‚Äî that's the model restating the type error), while AI findings elsewhere survive. Because they land in the same list, **every real type error inherits the existing "Fix with AI" action** ‚Üí a prefilled Cmd-K inline edit. Diagnostics run automatically (debounced 500ms, no tokens); hover and Ctrl/‚åò+Click / F12 go-to-definition live in `src/editor/lsp.ts`.
- **References & compiler-exact rename.** Shift+F12 opens an in-editor drawer of `findReferences` results (def/write badges, click navigates; peek-style ‚Äî a cross-file jump remounts the keyed editor and dismisses it). F2 on a TS/JS file asks the worker for `findRenameLocations` with `providePrefixAndSuffixTextForRename: false` (alias-renames like `import { double as twice }` would betray "rename across files"; safe because nothing applies silently) ‚Äî the worker applies spans against **its own snapshots** (`applyTextEdits` in `tsLsp.ts`, pure/unit-tested; refuses overlaps) and returns full new file contents, which `src/lib/rename.ts::buildRenamePlan` (pure, unit-tested; rejects out-of-root files) turns into a pre-built Composer plan ‚Üí the existing diff/checkpoint/apply pipeline. Non-TS files keep the model-mediated grep flow as fallback.
- **`lspClose` fires on tab close only (`FilePanel.closeTab`), never on file switch.** Every open tab must stay a program root: evicting the previous file on switch silently blinds cross-file queries (references/rename) to the files that import the one on screen ‚Äî diagnostics/hover/definition don't notice because they only need the *current* file's import closure. This regression is exactly what `lsp-verify`'s references stage exists to catch.
- **`linter(null)` in `FileEditor`'s extensions is load-bearing.** It installs `@codemirror/lint`'s state field (and the lint tooltip) up front. Without it, `setDiagnostics` takes its fallback path and appends the lint extensions via `appendConfig` *in the same transaction* that carries the diagnostics ‚Äî but a field added by `appendConfig` starts from `create()` and never sees that transaction's effects, so the first batch is silently dropped. `lintGutter()` has its own field, so the symptom is a gutter marker and a problem count **with no underline and no tooltip**.
- **Verify it for real:** `npm run lsp-verify` (Playwright on the built app) drives a genuine type error to a rendered squiggle, asserts the tooltip carries the compiler's message *and* the Fix-with-AI action, F12-navigates across files, Shift+F12-lists references across files, and lands an F2 rename as a two-file Composer plan.

### SwarmAgent desktop widget (second window)

A small frameless, transparent, always-on-top floating window that hosts just the SwarmAgent chat, so the assistant stays reachable when the main window is minimized or hidden to the tray. It is the **same renderer bundle**, loaded with a `#widget` hash ‚Äî `src/main.tsx` branches on `window.location.hash === '#widget'` and mounts `SwarmAgentWidget` instead of `App` (so none of App's heavy effects ‚Äî PTYs, conductor, loops ‚Äî run in the widget) and adds a `widget` class to `<html>` so the page is transparent and the card's rounded corners show. No extra Vite/electron-builder entry is needed.

- **Slim bar + dynamic height**: it starts as a small floating **bar** (just the input bar ‚Äî `createWidgetWindow()` opens at 420√ó70 near the bottom-right) and grows **upward** to ~460px to show a compact transcript once a conversation is on screen. The renderer drives this: it sends the target height on `widget:resize`, and main re-bounds the window anchored to its bottom edge (so it expands above the bar, never below).
- **Window/lifecycle** (`electron/main.ts`): `createWidgetWindow()` (frameless, `transparent`, `resizable: false`, `alwaysOnTop: 'floating'`, `skipTaskbar`). It auto-shows when the main window fires `minimize`/`hide` and auto-hides on `restore`/`show`/`focus` (unless the user dismissed it this session ‚Äî `widgetDismissed`). Tray menu gains **Show Chat Widget**. Dragging is pure CSS: the bar/header have `-webkit-app-region: drag` (buttons/inputs are `no-drag`), so the OS window moves around the desktop. The widget window also gets its own mic `setPermissionRequestHandler` for voice.
- **Appearance + voice**: `useAppearanceSync()` replays the persisted theme/accent/density/font settings into the widget's own document (it's a separate renderer), so the widget matches whatever the user picked. Voice input works via the same `useVoice` hook as the in-app chat (its own Whisper instance in this window).
- **Streaming** (`electron/ipc/swarmagent.ts`): `swarmAgent:chat` streams deltas to **`e.sender`** (the invoking window), so the widget ‚Äî not just the main window ‚Äî receives them.
- **Tool forwarding**: the widget runs its own `useSwarmAgent` loop but has **no workspace state of its own**, so it forwards each tool call. `useSwarmAgent({ runTool })` takes an injectable runner; the widget passes one that calls `window.swarmmind.widgetForwardTool(name,args)` ‚Üí main process `widget:forwardTool` relays it as a `widget:runTool` request to the **main window**, which runs the real `runTool` (via `useWidgetBridge`, mounted in `App`) against its store/PTYs and replies on `widget:toolResult` (correlated by id, 60s timeout). So "open the board" / "what changed?" from the widget actually drive the main app. Both windows share the same `localStorage['swarmagent:history']` (same `file://` origin), giving conversation continuity across surfaces.

### Startup performance ‚Äî lazy overlays

`App.tsx` eagerly imports only the always-visible shell (TopBar, CenterArea/terminal grid, StartScreen, sidebars). The on-demand center overlays ‚Äî **FilePanel (CodeMirror, ~975 KB), KanbanBoard, MemoryView, WorktreeReview, ComposerPanel, SwarmTimeline, ChangesPanel, CheckpointPanel, BenchmarksPanel, SwarmAgentChat, LoopsPanel** ‚Äî are `React.lazy(() => import(...).then(m => ({ default: m.X })))` (named exports ‚Üí default-wrapped) and rendered inside a single `<Suspense fallback={‚Ä¶}>` around the center conditional. This keeps them (and CodeMirror) out of the initial chunk: the renderer entry bundle is ~1.33 MB instead of ~2.54 MB, so first paint (which only needs the terminal grid) is faster. When adding a new center overlay, prefer lazy-loading it the same way unless it's part of the default view. (Whisper/`transformers` voice was already a separate lazy chunk.)

### Native modules

`node-pty` and `better-sqlite3` are native Node.js addons. They must be excluded from Vite bundling (see `electron.vite.config.ts` `externals`) and unpacked from asar in production (see `electron-builder.yml` `asarUnpack`). After upgrading Electron, run `npm run rebuild` to recompile them against the new Electron ABI.
